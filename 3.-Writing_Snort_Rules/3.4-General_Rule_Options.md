### 3.4.1 msg

msg 규칙 옵션은 패킷 덤프를 생성하고나 경고(alert)를 보낼 때 함께 출력할 메시지를 로깅 및 이벤트 엔진에 전달합니다. 이 옵션은 단순한 텍스트 문자열로 구성되지만, 세미콜론(`;`)처럼 Snort의 규칙 파서(Parser)를 혼란스럽게 할 수 있는 특정 문자를 포함할 때는 백슬래시(`\`)를 이스케이프 문자로 사용하여 해당 문자를 있는 그대로 표시합니다.

#### 3.4.1.1 Format

```snort
msg:"<message text>";
```

### 3.4.2 reference

참조(reference) 키워드는 룰에 외부 공격 식별 시스템에 대한 참조를 포함할 수 있게 합니다. 플러그인은 현재 여러 특정 시스템과 고유 URL을 지원합니다. 이 플러그인은 출력 플러그인이 생성된 알림에 대한 추가 정보를 제공하는 데 사용됩니다.

#### 지원 시스템
| System | URL Prefix |
| --- | --- |
| bugtraq | http://www.securityfocus.com/bid/ |
| cve | http://cve.mitre.org/cgi-bin/cvename.cgi?name= |
| mcafee | http://vil.nai.com/vil/content/v_ |
| osvdb | http://osvdb.org/show/osvdb/ |
| msb | http://technet.microsoft.com/en-us/security/bulletin/ |
| url | http:// |

#### 3.4.2.1 Format

```snort
reference:<id system>, <id>; [reference:<id system>, <id>;]
```

#### example)

```snort
alert tcp any any -> any 7070 (msg:"IDS411/dos-realaudio"; \
    flags:AP; content:"|fff4 fffd 06|"; reference:arachnids,IDS411;)
```

```snort
alert tcp any any -> any 21 (msg:"IDS287/ftp-wuftp260-venglin-linux"; \
    flags:AP; content:"31c031db 31c9b046 cd80 31c031db|"; \
    reference:arachnids,IDS287; reference:bugtraq,1387; \
    reference:cve,CAN-2000-1574;)
```

### 3.4.3 gid

gid(generator id) 키워드는 특정 규칙이 실행될 때 Snort의 어느 부분이 이벤트를 생성하는지 식별하는 데 사용됩니다. 예를 들어, gid 1은 룰 서브시스템과 연관되어 있고, 100 이상의 다양한 gid는 특정 프리프로세서서(Preprocessor)와 디코더(Decoder)에 할당되어 있습니다. 현재 사용 중인 generator ID 목록은 소스 트리의 `etc/generators` 파일에서 확인할 수 있습니다.

gid 키워드는 **선택사항**이며, 규칙에 명시되지 않으면 기본값인 `1`로 설정되어 일반 규칙 서브시스템의 일부가 됩니다. Snort 내부에 정의된(어떤 이유로 `etc/generators`에 누락된) gid와의 잠재적인 충돌을 방지하기 위해, 사용자 정의 시에는 **1,000,000 이상의 값**을 사용할 것을 권장합니다. 일반적인 규칙 작성 시에는 gid 키워드를 사용하지 않는 것이 좋으며, 이 옵션은 **반드시 sid 키워드와 함께 사용해야 합니다.** [(See section)](#344-sid)

`etc/gen-msg.map` 파일에는 프리프로세서와 디코더 gid에 관한 더 많은 정보가 포함되어 있습니다.

#### 3.4.3.1 Format

```snort
gid:<generator id>;
```

#### example)

이 예시는 generator id가 1000001인 규칙입니다.
```snort
alert tcp any any -> any 80 (content:"BOB"; gid:1000001; sid:1; rev:1;)
```

### 3.4.4 sid

이 sid 키워드는 Snort 룰을 고유하게 식별하는 데 사용됩니다. 이 정보는 플러그인이 룰을 쉽게 식별할 수 있게 해줍니다. 이 옵션은 rev 키워드와 함께 사용해야 합니다. [(See section)](#345-rev)

- < 100 : 미래 사용을 위해 예약(쓰지 않음)
- 100-999,999 : 스노트 배포에 포함된 
- \>= 1,000,000 : 로컬 룰셋에 사용됨

`sid-msg.map` 파일은 경고 메시지와 Snort 룰 ID의 매핑을 포함하고 있습니다. 이 정보는 ID를 alert 메시지에 매핑하기 위한 후처리 알림(post-processing alert)에 유용합니다.

#### 3.4.4.1 Format

```snort
sid:<snort rules id>;
```

#### example)

이 예시는 1000983의 Snort Rule ID를 가진 규칙입니다.
```snort
alert tcp any any -> any 80 (content:"BOB"; sid:1000983; rev:1;)
```

### 3.4.5 rev

*\* 쉽게 말해 버전 관리 번호*

이 rev 키워드는 Snort 룰의 리비전(revisions)을 고유하게 식별하는 데 사용됩니다. 리비전 번호는 Snort 룰 id(sid)와 함께 사용되어, 기존 시그니처나 설명을 보완하거나 최신 정보로 업데이트할 수 있도록 해줍니다. 이 옵션은 반드시 **sid 키워드와 함께 사용**해야 합니다. [(See section)](#344-sid)

#### 3.4.5.1 Format

```snort
rev:<revision integer>;
```

#### example)

이 예시는 Snort 룰 리비전 1 입니다.
```snort
alert tcp any any -> any 80 (content:"BOB"; sid:1000983; rev:1;)
```

### 3.4.6 classtype

classtype 키워드는 특정 규칙이 탐지하는 공격을 더 넓은 범위의 '공격 유형 클래스'의 일부로 분류하는 데 사용됩니다. Snort는 기본적으로 제공하는 룰 세트에서 활용할 수 있도록 표준 공격 클래스 집합을 미리 구성하요 제공하고 있습니다. 룰마다 이러한 분류(Classifications)를 정해두면, Snort가 생성하는 방대한 이벤트 데이터를 훨씬 체계적으로 정리하고 관리할 수 있습니다.

#### 3.4.6.1 Format

```snort
classtype:<class name>;
```

#### example)

```snort
alert tcp any any -> any 25 (msg:"SMTP expn root"; flags:A+; \
    content:"expn root"; nocase; classtype:attempted-recon;)
```

\
Snort가 정의한 공격 분류는 `classification.config` 파일에 있습니다. 이 파일은 다음과 같은 문법을 사용합니다.

```snort
config classification: <class name>,<class description>,<default priority>
```

\
이러한 공격 분류는 아래의 표에 있습니다. 현재 4가지 기본 우선순위(priority)로 정렬되어 있습니다. priority 1(high)이 가장 심각하고 4(very low)가 가장 적은 경우입니다.

| 분류 유형 (Classtype) | 설명 (Description) | 우선순위 (Priority) |
| --- | --- | --- |
| **attempted-admin** | 관리자 권한 획득 시도 | high |
| **attempted-user** | 일반 사용자 권한 획득 시도 | high |
| **inappropriate-content** | 부적절한 콘텐츠 감지 | high |
| **policy-violation** | 기업 보안 정책 위반 가능성 | high |
| **shellcode-detect** | 실행 가능한 코드(쉘코드) 탐지 | high |
| **successful-admin** | 관리자 권한 획득 성공 | high |
| **successful-user** | 일반 사용자 권한 획득 성공 | high |
| **trojan-activity** | 네트워크 트로이 목마 탐지 | high |
| **unsuccessful-user** | 일반 사용자 권한 획득 실패 | high |
| **web-application-attack** | 웹 애플리케이션 공격 | high |
| **attempted-dos** | 서비스 거부(DoS)공격 시도 | medium |
| **attempted-recon** | 정보 유출 시도(정찰 활동) | medium |
| **bad-unknown** | 잠재적으로 유해한 트래픽 | medium |
| **default-login-attempt** | 기본 계정 정보를 이용한 로그인 시도 | medium |
| **denial-of-service** | 서비스 거부(DoS) 공격 탐지 | medium |
| **misc-attack** | 기타 공격 | medium |
| **non-standard-protocol** | 비표준 프로토콜 또는 이벤트 탐지 | medium |
| **rpc-portmap-decode** | RPC 쿼리 디코딩 | medium |
| **successful-dos** | 서비스 거부(DoS) 성공 | medium |
| **successful-recon-largescale** | 대규모 정보 유출 | medium |
| **successful-recon-limited** | 정보 유출 | medium |
| **suspicious-filename-detect** | 의심스러운 파일 이름 탐지 | medium |
| **suspicious-login** | 의심스러운 계정명을 이용한 로그인 시도 탐지 | medium |
| **system-call-detect** | 시스템 콜 탐지 | medium |
| **unusual-client-port-connection** | 클라이언트의 비정상적인 포트 사용 | medium |
| **web-application-activity** | 취약한 웹 애플리케이션에 대한 접근 | medium |
| **icmp-event** | 일반적인 ICMP 이벤트 | low |
| **misc-activity** | 기타 활동 | low |
| **network-scan** | 네트워크 스캔 탐지 | low |
| **not-suspicious** | 의심스럽지 않은 트래픽 | low |
| **protocol-command-decode** | 일반 프로토콜 명령 디코딩 | low |
| **string-detect** | 의심스러운 문자열 탐지 | low |
| **unknown** | 알 수 없는 트래픽 | low |
| **tcp-connection** | TCP 연결 탐지 | very low |

#### 3.4.6.3 Warnings

classtype 옵션은 `snort.conf` 파일 내에서 `config classification` 설정을 통해 미리 정의된 분류(Classification)들만 사용할 수 있습니다. Snort는 기본적으로 제공하는 규칙들이 활용할 수 있도록 `classification.config`파일에 표준 분류 세트를 포함하여 제공하고 있습니다.

### 3.4.7 Priority

priority 태그는 룰에 심각도(위험도) 수준을 할당합니다. 기본적으로 classtype을 지정하면 해당 분류에 설정된 기본 우선순위가 자동으로 적용되지만, priority 옵션을 직접 사용하면 이 값을 덮어써서 재정의 할 수 있습니다. 각 경우에 대한 예시는 아래와 같습니다.
 
#### 3.4.7.1 Format 

```snort
priority:<priority integer>;
```

#### example)

```snort
alert tcp any any -> any 80 (msg:"WEB-MISC phf attempt"; flags:A+; \
    content:"/cgi-bin/phf"; priority:10;)
```

```snort
alert tcp any any -> any 80 (msg:"EXPLOIT ntpdx overflow"; \
    dsize:>128; classtype:attempted-admin; priority:10;)
```

### 3.4.8 metadata

meatadata 대그를 통해 룰 작성자는 규칙에 대한 추가 정보를 일반적으로 키-값(key-value) 형식으로 포함할 수 있습니다. 특정 metadata 키와 값은 Snort에서 특별한 의미를 가지며 이는 아래 표에 나열되어 있습니다. 표에 나열된 것 이외의 키는 Snort에서 사실상 무시되며, 키와 값으로 구성된 자유로운 형식을 가질 수 있습니다. 여러 키는 쉼표로 구분하며, 키와 값은 공백으로 구분합니다.

#### Table: Snort Metadata Keys

| 키 (Key) | 설명 (Description) | 값 형식 (Value Format) |
| --- | --- | --- |
| engine | 공유 라이브러리 규칙(Rule)을 나타냅니다. | "shared" |
| soid | 공유 라이브러리 룰 Generator 및 SID | gid\|sid |
| service | 대상 기반 서비스(Target-Based Service) 식별자 | "http" |


> 참고:
> service 메타데이터 키는 호스트 속성 테이블(Host Attribute Table)이 제공되는 경우에만 의미가 있습니다

#### 3.4.8.1 Format

아래 예시는 공유 라이브러리 규칙의 스텁(stub) 규칙을 보여줍니다. 첫 번째는 여러 개의 metadata 키워드를 사용하며, 두 번째는 쉼표로 구분된 키와 단일 metadata 키워드를 사용합니다.

```snort
metadata:key1 value1;
metadata:key1 value1, key2 value2;
```

#### example)

```snort
alert tcp any any -> any 80 (msg:"Shared Library Rule Example"; \
    metadata:engine shared; metadata:soid 3|12345;)

alert tcp any any -> any 80 (msg:"Shared Library Rule Example"; \
    metadata:engine shared, soid 3|12345;)

alert tcp any any -> any 80 (msg:"HTTP Service Rule Example"; \
    metadata:service http;)
```
