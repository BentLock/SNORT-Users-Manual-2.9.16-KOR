### 3.5.1 content

`content` 키워드는 Snort의 중요한 기능 중 하나입니다. 사용자는 이 키워드를 사용하여 패킷 페이로드에서 특정 content를 검색하고 해당 데이터에 따라 응답을 트리거하는 룰을 설정할 수 있습니다. content option pattern match가 수행될 때마다 Boyer-Moore 패턴 일치 함수가 호출되어 패킷 내용에 대해 (다소 계산 비용이 많이 드는) 테스트트가 수행됩니다. 인자 데이터 문자열과 정확히 일치하는 데이터가 패킷 페이로드의 어느 곳에든 포함되어 있으면 테스트가 성공하고 나머지 룰 옵션 테스트가 수행됩니다. 이 테스트는 대소문자를 구분한다는 점에 유의해야 합니다.

content 키워드에 대한 옵션 데이터는 다소 복잡하며, 텍스트와 바이너리 데이터가 혼합되어 포함될 수 있습니다. 바이너리 데이터는 일반적으로 파이프(`|`)로 묶여 바이트코드로 표현됩니다. 바이트코드는 바이너리 데이터를 16진수로 나타내며, 복잡한 바이너리 데이터를 간략하게 표현하는 좋은 방법 입니다. 아래 예시는 Snort 규칙에서 텍스트와 바이너리 데이터를 혼합하여 사용하는 방법을 보여줍니다.

하나의 룰에 여러 content 규칙을 지정할 수 있다는 점에 유의하십시오. 이를 통해 오탐을 줄이도록 룰을 맞춤 설정할 수 있습니다. *여러 content를 적용하여 AND 조건으로 룰을 작성할 수 있습니다.*

규칙 앞에 `!`이 붙으면, 해당 내용이 포함되지 않은 패킷에 대해 alert가 발생합니다. 이는 특정 패턴과 일치하지 않는 패킷에 대해 alert를 발생시키려는 규칙을 작성할 때 유용합니다.

> 참고:
>
> 또한 content 규칙 내에서 다음 문자는 이스케이프 처리해야 합니다.
>
> `;` `\` `"`

#### 3.5.1.1 Format

```snort
content:[!]"<content string>";
```

#### example)

```snort
alert tcp any any -> any 139 (content:"|5c 00|P|00|I|00|P|00|E|00 5c|";)

alert tcp any any -> any 80 (content:!"GET";)
```

> 참고:
>
> `!` 수식어는 함께 사용된 다른 수식어들의 조건을 모두 포함하여, 해당 콘텐츠 검색 결과 전체를 반전(부정)시킵니다. 예를 들어, `content:!"A"; within:50;` 옵션을 사용했는데 실제 패킷의 페이로드가 5바이트 뿐이고 그 안에 "A"가 없다면, Snort는 이를 조건에 부합하는 것(Match)으로 판단합니다. 만약 50바이트 이상의 데이터가 존재할 때만 이 규칙이 유효하게 작동하길 원한다면, `content` 옵션 앞단에 `isdataat`을 배치하여 데이터 길이를 먼저 검증해야 합니다.
>
> *\* 보충 설명 : `within:50`은 "지금부터 50바이트 안에서 찾아라" 라는 뜻입니다.*
> *그런데 패킷이 5바이트밖에 안 되면, 당연히 그 안에서 "A"를 찾을 수 없습니다. `!`가 붙어있으니 "A가 없으니 탐지"가 되어버려 오탐이 발생할 수 있습니다.*
> *그래서 `isdataat:50; content:!"A"; within:50;` 처럼 작성하여, 일단 50바이트는 되는지 보고, 그 다음에 A가 없는지 확인하는 식으로 순서를 잡아주는 것이 중요합니다.*

### 3.5.2 protected_content

`protected_content` 키워드는 `content` 키워드와 유사한 기능을 제공하지만, 작동 방식과 활용법은 매우 다릅니다. content와의 가장 큰 차별점은 탐지하려는 원본 데이터를 직접 드러내지 않고, 해당 데이터의 보안 해시 다이제스트(Secure Hash Digest)만을 공개하여 대상을 숨길 수 있다는 것입니다.

`content` 키워드와 마찬가지로 특정 바이트 문자열을 매칭하는 것이 주된 목적이지만, 탐지 과정에서 유입되는 패킷의 일부분을 일일이 해싱(Hashing)한 뒤 제공된 해시값과 비교하는 방식을 취합니다. 이 때문에 일반적인 문자열 매칭보다 계산 비용이 많이 듭니다.

현재 `protected_content` 키워드와 함께 `MD5`, `SHA256` 및 `SHA512` 해시 알고리즘을 사용할 수 있습니다. Snort 구성에서 기본값이 설정되지 않은 경우 규칙에서 `hash`를 사용하여 해시 알고리즘을 지정해야 합니다. 또한 원시 데이터의 길이를 나타내기 위해 `protected` 앞에 `length` 수정자를 지정해야 합니다.

`content`와 마찬가지로 하나의 규칙에 여러 개의 `protected_content` 규칙을 사용할 수 있습니다. 또한 여러 개의 `protected_conent` 규칙과 여러 개의 `content` 규칙을 함께 하용할 수도 있습니다.

규칙 앞에 `!`이 붙으면 대상 `content`가 포함되지 않은 패킷에 대해 경고가 발생합니다. 이는 특정 패턴과 일치하지 않는 패킷에 대해 alert를 발생시키려는 룰을 작성할 때 유용합니다.

> 참고:
>
> `protected_content` 키워드는 일부(전부는 아님) content 수정자와 함께 사용할 수 있습니다. 지원되지 않는 수정자는 다음과 같습니다.
>
> ```text
> nocase
> fast_pattern
> depth
> within
> ```

#### 3.5.2.1 Format

```snort
protected_content:[!]"<content hash>", length:orig_len[, hash:md5|sha256|sha512];
```

#### example)

"HTTP" 문자열에 대한 alert:

```snort
alert tcp any any <> any 80 (msg:"MD5 Alert"; \
    protected_content:"293C9EA246FF9985DC6F62A650F78986"; hash:md5; offset:0; length:4;)
```

```snort
alert tcp any any <> any 80 (msg:"SHA256 Alert"; \
    protected_content:"56D6F32151AD8474F40D7B939C2161EE2BBF10023F4AF1DBB3E13260EBDC6342"; hash:sha256; offset:0; length:4;)
```

> 참고:
>
> `!` 수식어는 함께 사용된 다른 수식어들의 조건을 모두 포함하여, 해당 콘텐츠 검색 결과 전체를 반전(부정)시킵니다. 예를 들어, `content:!"A"; within:50;` 옵션을 사용했는데 실제 패킷의 페이로드가 5바이트 뿐이고 그 안에 "A"가 없다면, Snort는 이를 조건에 부합하는 것(Match)으로 판단합니다. 만약 50바이트 이상의 데이터가 존재할 때만 이 규칙이 유효하게 작동하길 원한다면, `content` 옵션 앞단에 `isdataat`을 배치하여 데이터 길이를 먼저 검증해야 합니다.
>
> *\* 보충 설명 : `within:50`은 "지금부터 50바이트 안에서 찾아라" 라는 뜻입니다.*
> *그런데 패킷이 5바이트밖에 안 되면, 당연히 그 안에서 "A"를 찾을 수 없습니다. `!`가 붙어있으니 "A가 없으니 탐지"가 되어버려 오탐이 발생할 수 있습니다.*
> *그래서 `isdataat:50; content:!"A"; within:50;` 처럼 작성하여, 일단 50바이트는 되는지 보고, 그 다음에 A가 없는지 확인하는 식으로 순서를 잡아주는 것이 중요합니다.*

### 3.5.3 hash

`hash` 키워드는 `protected_content` 규칙을 일치시킬 때 사용할 해싱 알고리즘을 지정하는 데 사용됩니다. Snort 구성에서 기본 알고리즘이 지정되지 않은 경우, `Protected_contet` 규칙에서 사용할 알고리즘을 지정해야 합니다. 현재 `MD5`, `SHA256`및 `SHA512`가 지원됩니다.

#### 3.5.3.1 Format

```snort
hash:[md5|sha256|sha512];
```

### 3.5.4 length

`length` 키워드는 `protected_content` 규칙의 해시값(digest)으로 지정된 원본 콘텐츠의 길제 길이를 명시하는 데 사용됩니다. 설정값은 반드시 0보다 크고 `65536`(=2^16)보다 작은 값이어야 합니다.

#### 3.5.4.1 Format

```snort
length:[<original_length>];
```

### 3.5.5 nocase

`nocase` 키워드를 사용하면 룰 작성자가 Snort가 어떤 형식으로 출력해야 하는지 지정할 수 있습니다. **대소문자를 무시하고 특정 패턴을 지정합니다.** `nocase`는 직전에 명시된된 `content` 옵션에 적용되는 수식어 역할을 합니다. *이는 여러 개의 content키워드를 사용할 때, nocase 키워드는 바로 앞의 content 키워드에 영향을 준다는 의미이기도 합니다.*

#### 3.5.5.1 Format

```snort
nocase;
```

#### example

```snort
alert tcp any any -> any 21 (msg:"FTP ROOT"; content:"USER root"; nocase;)
```

### 3.5.6 rawbytes

`rawbytes` 키워드는 전처리기(Preprocessor)에 의해 수행된 모든 디코딩 과정을 무시하고, 룰이 패킷의 원본 데이터(raw data)를 직접 검사할 수 있게 해줍니다. 이 키워드는 직전에 명시된 `content` 옵션에 적용되는 수식어 역할을 합니다.

HTTP Inspect 전처리기는 `http_raw_cookie`, `http_raw_header`, `http_raw_uri` 등과 같이 가공되지 않은 데이터를 다루는 전용 키워드들을 제공하며, 이를 통해 HTTP 요청 및 응답의 특정 부분에 대한 원본 데이터를 매칭할 수 있습니다.

그 외 대부분의 전처리기들은 `rawbytes`를 명시적으로 지정하지 않는 한, 기본적으로 디코딩되거나 정규화(Normalized)된 데이터를 콘텐츠 매칭에 사용합니다. 따라서 패킷 내의 가공되지 않은 임의의 원본 데이터를 직접 검사하려면 반드시 `rawbytes`를 명시해야 합니다.

#### 3.5.6.1 Format

```snort
rawbytes;
```

#### example)

이 예시는 content 패턴 매처에게 Telnet 디코더가 제공하는 디코딩된 트래픽 대신 원시 트래픽을 살펴보라고 지시합니다.

```snort
alert tcp any any -> any 21 (msg:"Telnet NOP"; content:"|FF F1|"; rawbytes;)
```

### 3.5.7 depth

`depth` 키워드를 사용하면 룰 작성자가 Snort가 패킷에서 지정된 패턴을 검색할 위치를 지정할 수 있습니다. `depth`는 룰에서 이전에 있는 `content` 키워드를 수정합니다.
`depth` 값을 5로 설정하면 Snort는 페이로드의 처음 5바이트 내에서만 지정된 패킷을 찾습니다.

`depth` 키워드는 이전 `content` 키워드에 대한 수정자이므로 `depth`가 지정되기 전에 룰에 `cnotent`가 반드시 있어야 합니다.

이 키워드는 검색하려는 패턴 길이보다 크거나 같은 값을 허용합니다. 최소 허용 값은 `1`이고, 최대 허용 값은 `65535`입니다.

값(value)은 동일한 규칙 내에서 `byte_extract` 키워드를 통해 추출한 변수명을 참조하는 문자열로 설정할 수도 있습니다. *depth, offset, within, distance, isdataat 등에 고정된 숫자 외에 변수 이름을 넣을 수 있다는 의미입니다.*

#### 3.5.7.1 Format

```snort
depth:[<number>|<var_name>];
```

### 3.5.8 offset

`offset` 키워드를 사용하면 룰 작성자가 패킷 내에서 패턴을 검색하기 시작할 위치를 지정할 수 있습니다. `offset`은 룰에서 이전의 `content` 키워드를 수정합니다.

`offset` 값을 5로 설정하면 Snort는 페이로드의 처음 5바이트 이후부터 지정된 패턴을 찾기 시작합니다.

이 키워드는 이전 `content` 키워드에 대한 수정자이므로 `offset`이 지정되기 전에 룰에 `content`가 반드시 있어야 합니다.

이 키워드는 `-65535`부터 `65535`까지의 값을 허용합니다.

값(value)은 `byte_extract` 키워드를 통해 추출한 변수명을 참조하는 문자열로 설정할 수도 있습니다. *depth, offset, within, distance, isdataat 등에 고정된 숫자 외에 변수 이름을 넣을 수 있다는 의미입니다.*

#### 3.5.8.1 Format

```snort
offset:[<number>|<var_name>];
```

#### example)

다음 예시는 `content`, `offset` 및 `depth`검색 규칙을 결합하여 사용하는 방법을 보여줍니다.

```snort
alert tcp any any -> any 80 (content:"cgi-bin/phf"; offset:4; depth:20;)
```

### 3.5.9 distance

`distance` 키워드를 사용하면 룰 작성자가 Snort가 이전 패턴 매칭이 끝난 지점을 기준으로, 다음 패턴 검색을 시작하기 전까지 무시할 바이트 수를 지정합니다.

이는 패킷의 시작 지점을 기준으로 하는 [`offset`](#358-offset)과 기능적으로는 동일하지만, 기준점이 패킷의 맨 앞이 아니라 **마지막 패턴 매칭이 끝난 위치**라는 점이 다릅니다.

이 키워드는 `-65535`부터 `65535`까지의 값을 허용합니다.

값(value)은 `byte_extract` 키워드를 통해 추출한 변수명을 참조하는 문자열로 설정할 수도 있습니다. *depth, offset, within, distance, isdataat 등에 고정된 숫자 외에 변수 이름을 넣을 수 있다는 의미입니다.*

#### 3.5.9.1 Format

```snort
distance:[<byte_count>|<var_name>];
```

#### example)

아래 룰은 `/ABC.{1,}DEF/` 정규표현식에 해당합니다.

```snort
alert tcp any any -> any any (content:"ABC"; content:"DEF"; distance:1;)
```

### 3.5.10 within

`within` 키워드는 `content` 키워드를 이용한 패턴 매칭 시, 이전 매칭 지점으로부터 **최대 N바이트 이내**에서 다음 패턴을 찾도록 제한하는 `content` 수식어입니다. 이 옵션은 주로 `distance` 룰 옵션과 함께 결합하여 사용하도록 설계되었습니다.

이 키워드에는 검색하려는 패턴의 길이보다 크거나 같은 값을 지정해야 합니다. 또한, 이 키워드에 허용되는 최댓값은 `65535`입니다.

값(value)은 `byte_extract` 키워드를 통해 추출한 변수명을 참조하는 문자열로 설정할 수도 있습니다. *depth, offset, within, distance, isdataat 등에 고정된 숫자 외에 변수 이름을 넣을 수 있다는 의미입니다.*

#### 3.5.10.1 Format

```snort
within:[<byte_counte>|<var_name>];
```

#### example 1)

이 룰은 EFG 검색 범위를 ABC 매치 지점으로부터 10바이트 이상 벗어나지 않도록 제한합니다.

*아래 룰은 `/ABC.{,7}DEF/` 정규표현식에 해당합니다. `within`은 '끝나야 하는 지점을 제한하기 때문에 유의해야 합니다.*
```snort
alert tcp any any -> any any (content:"ABC"; content:"EFG"; within:10;)
```

#### example 2)

만약 '간격이 딱 10바이트 여야 한다'라는 조건을 걸고 싶으면 아래와 같이 `distance`와 `within`을 함께 사용하면 됩니다.

정규표현식으로 `/ABC.{10}EFG/`와 동일합니다.

```snort
alert tcp any any -> any any (content:"ABC"; content:"EFG"; distance:10; within:13;)
```

> 참고:
>
> 장비의 계산 비용을 고려하면 pcre(정규표현식) 보다 distance, within 조합이 더 효율적입니다.

### 3.5.11 http_client_body

`http_client_body` 키워드는 HTTP 클라이언트 요청 본문(body)으로 검색 범위를 제한하는 `content` 수정자입니다.

이 키워드는 이전 `content` 키워드에 대한 수정자이므로 `http_client_body`를 지정하기 전에 content가 있어야 합니다.

이 옵션으로 검사되는 데이터의 양은 HttpInspect의 `post_depth` 설정값에 따라 달라집니다. 만약 `post_depth`가 `-1`로 설정되어 있다면, 이 키워드를 사용한 패턴 매칭은 작동하지 않습니다.

#### 3.5.11.1 Format

```snort
http_client_body;
```

#### example)

이 룰은 "EFG"패턴에 대한 검색 범위를 HTTP 클라이언트 요청의 본문으로 제한합니다.

```snort
alert tcp any any -> any 80 (content:"ABC"; content:"EFG"; http_client_body;)
```

> 참고:
> 
> 동일한 content에 대해 `http_client_body` 수정자와 `rawbytes` 수정자를 함께 사용할 수 없습니다.

### 3.5.12 http_cookie

`http_cookie` 키워드는 검색 범위를 HTTP 클라이언트 요청이나 서버 응답에서 추출된 '쿠키 헤더 필드'로 제한하는 `content` 수식어입니다. (이 동작은 HttpInspect 설정에 따릅니다.)

쿠키 버퍼에는 헤더 이름(`Cookie:` 또는 `Set-Cookie:`)이나 앞부분의 공백, 그리고 줄 끝의 CRLF(줄바꿈 문자)가 포함되지 않습니다. 이러한 요소들은 일반적인 HTTP 헤더 버퍼에 포함됩니다.

이 키워드는 직전의 `content` 키워드를 수식하는 역할을 하므로, 반드시 `http_cookie`를 명시하기 전에 규칙 내에 `content`가 먼저 선언되어야 합니다.

또한, 이 키워드는 `enable_cookie` 설정 옵션에 의존합니다. 쿠키 헤더 필드는 이 옵션이 설정되었을 때만 별도로 추출됩니다. 만약 `enable_cookie`를 설정하지 않으면 쿠키 데이터는 일반 HTTP 헤더에 그대로 남게 되며, 이 경우 `http_cookie`를 사용하는 것은 `http_header`를 사용하는 것과 결과가 같습니다.

추출된 쿠키 헤더 필드는 HttpInspect 설정에 따라 정규화(NOMALIZED) 처리될 수 있습니다. [(참조: 2.2.7 HTTP Inspect)](http://manual-snort-org.s3-website-us-east-1.amazonaws.com/node17.html#sub:http-inspect)

| 구분 | HTTP 헤더 버퍼 예시 | 쿠키 전용 버퍼(`http_cookie`) |
| --- | --- | --- |
| 실제 데이터 | `Cookie: sessionid=1234\r\n` | `sessionid:1234` |
| 포함 항목 | 헤더이름, 공백, CRLF 포함 | **순수 데이터만 포함** |

#### 3.5.12.1 Formay

```snort
http_cookie;
```

#### example

이 룰은 HTTP 클라이언트 요청에서 추출된 쿠키 헤더 필드 내에서만 "EFG" 패턴을 검색하도록 제한합니다.

```snort
alert tcp any any -> any 80 (content:"ABC"; content:"EFG"; http_cookie;)
```

> 참고:
>
> `http_cookie` 수정자는 동일 `content`에 대한 `rawbytes`, `fast_pattern` 수정자와 함께 사용할 수 없습니다.

### 3.5.13 http_raw_cookie

`http_raw_cookie` 키워드는 HTTP 클라이언트 요청 또는 HTTP 서버 응답에서 추출된 `UNNORMALIZED Cookie Header` 필드로 검색을 제한하는 `content` 수정자입니다. [(HttpInspect의 구성에 따름)](http://manual-snort-org.s3-website-us-east-1.amazonaws.com/node17.html#sub:http-inspect)

이 키워드는 이전 `content` 키워드의 수정자이므로 `http_raw_cookie`가 지정되기 이전에 `content`가 있어야 합니다. `enable_cookie` 구성 옵션에 따라 달라집니다. 쿠키 헤더 필드는 이 옵션이 구성된 경우에만 추출됩니다.

#### 3.5.13.1 Format

```snort
http_raw_cookie;
```

#### example)

이 룰은 HTTP 클라이언트 요청에서 추출된 비정규화 쿠키 헤더 필드 내에서만 "EFG" 패턴을 검색하도록 제한합니다.

```snort
alert tcp any any -> any 80 (content:"ABC"; content:"EFG"; http_raw_cookie;)
```

> 참고:
>
> `http_raw_cookie` 수정자는 다음 항목과 함께 사용할 수 없습니다. 동일한 `content`에 대한 `rawbytes`, `http_cookie` 또는 `fast_pattern` 수정자.

### 3.5.14 http_header

`http_header` 키워드는 HTTP 클라이언트 요청 또는 HTTP 서버 응답의 추출된 헤더 필드로 검색을 제한하는 `content` 수정자입니다. [(HttpInspect의 구성에 따름)](http://manual-snort-org.s3-website-us-east-1.amazonaws.com/node17.html#sub:http-inspect)

이 키워드는 이전 `content` 키워드에 대한 수정자이므로 `http_header`가 지정되기 이전에 `content`가 있어야 합니다.

추출된 헤더 필드는 HttpInspect 설정에 따라 정규화(NOMALIZED) 처리될 수 있습니다. [(참조: 2.2.7 HTTP Inspect)](http://manual-snort-org.s3-website-us-east-1.amazonaws.com/node17.html#sub:http-inspect)

#### 3.5.14.1 Format

```snort
http_header;
```

#### example)

이 룰은 "EFG" 패턴 검색 범위를 HTTP 클라이언트 요청 또는 HTTP 서버 응답에서 추출된 헤더 필드로 제한합니다.

```snort
alert tcp any any -> any 80 (content:"ABC"; content:"EFG"; http_header;)
```

> 참고:
>
> `http_headeer` 수정자는 동일한 `content`에 대해 `rawbytes` 수정자를 함께 사용할 수 없습니다.
